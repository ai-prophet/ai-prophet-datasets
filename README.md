# ai-prophet-datasets

Standardized dataset artifacts and registry for `mini-prophet` eval.

## Repository layout

```text
datasets/<dataset>/dataset.json
datasets/<dataset>/releases/<release_id>/release.json
datasets/<dataset>/releases/<release_id>/tasks.jsonl
registry.json
```

A **dataset** is a named series (e.g. `prophet-recent`, `hackathon-day`).
A **release** is one fetchable batch of tasks under a dataset, identified
by a stable `release_id` (a date for time-based releases like
`2026-03-01`, or semver like `v1.0.0`). A **task** is one row inside
`tasks.jsonl`, identified by a `task_id` unique within its release.

### `dataset.json`

```json
{
  "name": "dummy",
  "description": "Dummy benchmark dataset"
}
```

### `release.json`

```json
{
  "release_id": "2026-03-01",
  "release_date": "2026-03-01",
  "description": "Release of forecasting tasks for 2026-03-01",
  "status": "open"
}
```

`status` is one of `open` (more resolutions may still arrive),
`closed` (resolutions finalized), or `archived` (kept for history,
not for new evals).

### `tasks.jsonl`

One JSON object per line. Required fields:

- `task_id` — non-empty string, unique within the release
- `title` — non-empty string
- `outcomes` — non-empty list of non-empty strings

Optional fields include `source`, `context`, `metadata`, and
`resolved_outcome`. The resolved outcome, when present, is structured:

```json
"resolved_outcome": {
  "value": ["Orlando"],
  "resolved_at": "2026-02-24T00:07:09Z",
  "source": "KXNBAGAME-26FEB22ORLLAC"
}
```

`value` is **always a list of strings**, even when only one outcome
resolved. Each entry must be drawn from the task's `outcomes` and
entries must be unique. The list shape is the canonical encoding —
single-outcome resolutions are still wrapped (`["Yes"]`, not `"Yes"`)
so consumers don't have to branch on type. Multi-entry lists express
multi-correct or partially-resolved questions (e.g. `["A", "C"]` for a
"select all that apply" task). If `resolved_outcome` is absent or
`null`, the task is unresolved.

## Registry

`registry.json` is a denormalized index of the tree, regenerated by CI on
every push to `main`. It exists for fast listing and discovery; it is
**never the source of truth** and should not be hand-edited.

```json
{
  "datasets": [
    {
      "name": "prophet-recent",
      "description": "...",
      "releases": [
        {
          "id": "v1.0.0",
          "release_date": "2026-02-24",
          "status": "closed",
          "task_count": 83,
          "resolved_count": 31,
          "path": "datasets/prophet-recent/releases/v1.0.0/tasks.jsonl"
        }
      ]
    }
  ]
}
```

## Design philosophy

A few choices are easier to live with once the reasoning is on the page.

**A release has a frozen event set; only resolutions mutate.** Once a
release ships, `task_id`/`title`/`outcomes` and the rest of the
descriptive fields are fixed. The single mutable field per task is
`resolved_outcome`, attached as events resolve over time. Adding,
removing, or rewording an event is not an edit — it is a new release
under a new `release_id`. This keeps every release citeable: a consumer
who fetched `prophet-recent/v1.0.0` last week and again today sees the
same task set, with possibly more `resolved_outcome` fields populated.

**Git is the audit log.** We deliberately do not store per-release
checksums or commit pins inside the registry — the repo is already
git-tracked, so every mutation is a normal commit visible in `git log`.
If a consumer needs a fully reproducible snapshot for an eval run, they
pin the repo to a commit sha at fetch time. Embedding that information
into the registry would just be a worse copy of what git already gives
us, and would have to be rewritten on every resolution edit.

**The registry is a derived index, not a database.** `registry.json` is
regenerated from the tree on every push. If it ever drifts, the fix is
to rebuild it; nothing downstream relies on registry-only state.
Validation logic lives in one place (`scripts/registry_manager.py`) and
is invoked by both CI and ad-hoc local runs.

**A "release" is a unit of fetching, not a derivative of a previous
release.** Two releases under the same dataset (e.g. consecutive
hackathon days) are not assumed to share lineage — they may have
entirely disjoint task sets. The dataset is the namespace; releases are
independent batches inside it. We avoid a `latest` pointer in
`dataset.json` for the same reason: "latest" is a query (sort releases
by `release_date`), not state worth storing.

## Maintainer workflow

1. Create the release directory: `datasets/<name>/releases/<release_id>/`.
2. Write `release.json` and `tasks.jsonl`.
3. (For a brand-new dataset only) write `datasets/<name>/dataset.json`.
4. Open a PR. CI runs validation against the entire tree.
5. Merge. On push to `main`, CI rebuilds and commits `registry.json`.

To attach a resolved outcome to an existing task, edit only that row's
`resolved_outcome` field in the existing `tasks.jsonl` and open a PR.

## Local commands

```bash
# Validate every dataset and release in the tree
python3 scripts/registry_manager.py validate --all

# Validate a single release directory
python3 scripts/registry_manager.py validate \
  --release datasets/dummy/releases/2026-03-01

# Rebuild registry.json from the tree
python3 scripts/registry_manager.py rebuild-registry
```

## CI automation

Workflow: `.github/workflows/dataset-registry-sync.yml`

- `pull_request`: validate all datasets and releases.
- `push` to `main`: validate, rebuild `registry.json`, commit if it changed.
